# 검증 루브릭 (6단계)

완성된 HTML을 아래 4영역 + **선택된 패턴 문서 끝의 '검증 추가 포인트'**로 점검한다. **전 항목 통과 전에는 교사에게 반환하지 않는다.** 불합격 항목은 수정 후 해당 항목만 재검증한다.

게임 구조상 적용 불가능한 항목은 **'해당 없음(N/A)'으로 판정하고 사유를 기록**한다 (예: 시간제 순환 출제 게임의 B2). N/A 항목은 통과로 간주하되 사유 없는 N/A는 허용하지 않는다.

## 점검 방법

1. **자체 검증(필수)**: 생성한 코드를 처음부터 다시 읽으며 각 항목을 판정한다. 머릿속 시뮬레이션이 아니라 실제 코드 라인을 근거로 판정한다 (예: "A2 통과 — questions 배열 12개 문항의 answer 인덱스를 자료와 대조함").
2. **브라우저 테스트(도구 가용 시)**: Playwright 등 브라우저 도구가 있으면 ① 파일 열기 ② 콘솔 오류 0건 확인 ③ 시작→문항 2개 이상 진행(정답·오답 각 1회)→결과 화면 도달 ④ 모바일(390px)·데스크톱(1280px) 두 폭에서 스크린샷 확인. 도구가 없으면 생략하고 교사 안내에 직접 확인 권유를 포함한다.

## A. 학습 내용 정확성 (가장 중요)

| # | 항목 | 판정 기준 |
|---|------|----------|
| A1 | 출제 범위 일치 | 모든 문항이 교사가 준 학습자료 안에서 출제됨. 자료에 없는 내용 지어내기 없음 |
| A2 | 정답 정확성 | 모든 문항의 정답을 자료와 1:1 대조 — 보기형은 answer 인덱스, 단답·교사판정형은 정답 텍스트 기준. 보기 섞기 로직이 있으면 섞은 후에도 정답 매핑이 유지되는지 코드로 확인 |
| A3 | 해설 정확성 | 오답 시 표시되는 해설이 사실에 맞고 학습자료와 모순 없음 |
| A4 | 수준 적합성 | 어휘·문장 길이·난이도가 해당 학년 수준에 맞음 (초1에 한자어 정의 금지 등) |
| A5 | 오개념 방지 | 게임 규칙이 잘못된 연합을 만들지 않음 (예: 틀린 보기를 반복 노출해 각인시키는 구조 금지, 짝맞추기에서 잘못된 짝이 '맞음'으로 처리될 가능성 없음) |

## B. 기능 완결성

| # | 항목 | 판정 기준 |
|---|------|----------|
| B1 | 전체 흐름 | 시작 → 모든 문항 진행 → 결과 화면까지 끊김 없이 도달 가능 |
| B2 | 마지막 문항 처리 | 마지막 문항 후 정상적으로 결과 화면 전환 (off-by-one 없음) |
| B3 | 다시 하기 | 결과 화면의 다시 하기가 점수·진행 상태·타이머를 완전히 초기화 |
| B4 | 연타·중복 입력 | 보기 연속 클릭, 피드백 표시 중 클릭 등에서 중복 채점·화면 깨짐 없음 |
| B5 | 경계값 | 0점 결과, 만점 결과, (타이머 게임) 시간 종료 시점 모두 정상 처리 |
| B6 | 오류 없는 코드 | 문법 오류, 미정의 변수·함수 없음. GAME_DATA의 모든 필드가 실제로 코드에서 읽는 이름과 일치 |
| B7 | 표시/숨김 정합성 | 스타일시트에 `display:none`인 요소를 `style.display = ""`로 복원하는 코드 없음 (클래스 토글로 통일). 각 화면 상태에서 진행에 필요한 버튼이 실제로 보이는지 상태별로 추적 |

## C. 기기·환경 대응

| # | 항목 | 판정 기준 |
|---|------|----------|
| C1 | 제로 의존 | 파일 내 `http` 검색 시 구글 폰트 도메인 외 외부 요청 없음. 폰트 폴백 지정됨 |
| C2 | 단일 파일 | CSS/JS 전부 인라인, 외부 파일 참조 없음 |
| C3 | 반응형 | 모바일 세로(360~430px)와 데스크톱(1280px+)에서 레이아웃·글자 크기 모두 사용 가능 |
| C4 | 터치 대응 | 탭 하이라이트 제거, 더블탭 줌 방지, 클릭·터치 모두 동작 |
| C5 | 저장 안전성 | localStorage 사용부가 try/catch로 보호됨 (미사용이면 통과) |

## D. 활용 방식·설계 적합성

| # | 항목 | 판정 기준 |
|---|------|----------|
| D1 | 명세서 일치 | 4단계에서 교사가 승인한 명세서(유형·문항 수·팀 수·규칙)와 산출물이 일치 |
| D2 | 주 활용 방식 최적화 | design-guide.md 5절의 해당 방식 요건 충족 (TV: 글자 크기·교사 조작 / 개별: 모바일 우선 / 전자칠판: 대형 터치 타겟) |
| D3 | 학습 피드백 | 오답 시 정답+해설 표시, 결과 화면에 틀린 문항 다시 보기 존재 |
| D4 | 데이터 분리 | GAME_DATA가 "문항 수정 구역" 주석과 함께 스크립트 최상단에 분리됨 |
| D5 | 디자인 톤 | 학교급별 톤 가이드 적용 (고등학생용에 유아적 문구 없음 등) |

## 보고 형식

검증 후 교사에게는 한 줄 요약만 보고한다:
> "완성도 검증을 마쳤어요 — 정답·해설 전수 대조, 처음부터 끝까지 동작 흐름, 모바일/큰 화면 대응까지 ○개 항목을 점검했고, 수정 ○건을 반영했습니다."

내부적으로는 항목별 통과/수정 내역을 기록해 두고, 교사가 상세를 물으면 보여준다.
